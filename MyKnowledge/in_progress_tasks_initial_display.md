# Особенность отображения задач в процессе на HR-дашборде

**Дата**: 13 мая 2025 г. (актуализировано)
**Категория**: Фронтенд/UI, WebSocket

## Описание проблемы

В приложении OnboardPro на странице HR-дашборда наблюдается следующая особенность поведения: при первоначальной загрузке страницы в блоке "Задачи в процессе" сначала отображается значение 0, а затем, после получения данных через WebSocket, отображается актуальное число задач.

## Решенная проблема

Ранее была исправлена основная проблема, связанная с несоответствием между счетчиком задач в процессе и фактическим списком этих задач. Причина заключалась в том, что на бэкенде количество задач в процессе определялось двумя разными способами:

1. Через прямой запрос `in_progress_task_query.count()`
2. Через подсчет элементов в списке детализации `len(in_progress_tasks_details)`

Решение заключалось в консистентном использовании второго значения (количество элементов в списке детализации) для обеспечения согласованности между счетчиком и списком задач.

## Текущее поведение

После исправления основной проблемы осталась следующая особенность:

1. Когда пользователь впервые загружает страницу HR-дашборда, блок "Задачи в процессе" изначально показывает значение 0.
2. После установления WebSocket-соединения и получения актуальных данных с сервера, значение обновляется на корректное количество задач.

Данное поведение не является критичным для работы приложения, но может быть улучшено в будущих обновлениях.

## Техническое объяснение

Первоначальное отображение нуля происходит из-за следующих факторов:

1. При инициализации компонента во фронтенде используются начальные значения (0) для счетчиков, пока не будут получены актуальные данные с сервера.
2. Соединение WebSocket устанавливается асинхронно после монтирования компонента.
3. Данные приходят только после успешного подключения и аутентификации WebSocket.

## Влияние на пользователя

Это поведение визуально заметно для пользователей как "мигание" значения счетчика задач при загрузке страницы. Однако это не влияет на функциональность системы и не приводит к неправильным действиям пользователя.

## Возможные улучшения

Хотя данная проблема не требует немедленного исправления, для улучшения пользовательского опыта в будущем можно рассмотреть следующие подходы:

1. Добавить состояние загрузки (скелетон или спиннер) для блоков со статистикой вместо отображения нулей.
2. При инициализации страницы выполнять обычный REST API запрос для получения начальных данных, а затем обновлять их через WebSocket.
3. Кэшировать последние известные значения в localStorage и восстанавливать их при загрузке страницы до получения свежих данных.

## Связанные компоненты

- `/frontend/src/pages/HRDashboard.jsx` - компонент дашборда HR
- `/frontend/src/components/common/WebSocketProvider.jsx` - провайдер WebSocket соединения
- `/backend/main.py` - API эндпоинты и WebSocket обработчик на бэкенде
- `/backend/websocket_manager.py` - менеджер WebSocket соединений

## Заключение

Данная особенность не является критической ошибкой, а представляет собой обычное поведение в асинхронных веб-приложениях. Пользователи быстро получают актуальные данные, и визуальный эффект "мигания" счетчика минимален.
