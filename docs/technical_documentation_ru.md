## Архитектура системы

OnboardPro следует современной архитектуре веб-приложения:

- **Backend**: FastAPI (Python) с базой данных SQLite и SQLAlchemy ORM
- **Frontend**: React с Tailwind CSS и Heroicons
- **Инфраструктура**: Docker и docker-compose для контейнеризации
- **Аутентификация**: JWT-based система аутентификации
- **Внешние интеграции**: Telegram Bot, Google Calendar, Workable API
- **Миграции**: Alembic для управления версиями базы данных

## Среда разработки

### Предварительные требования

- Docker и Docker Compose
- Node.js (v18+) и npm
- Python 3.11+

### Инструкции по установке

1. Клонировать репозиторий
2. Создать файл `.env` с необходимыми переменными окружения
3. Запустить `docker-compose up -d` для старта контейнеров
4. Открыть приложение по адресу http://localhost:3000

### Переменные окружения

Необходимые переменные окружения:

- `DATABASE_URL`: Путь к базе данных SQLite (по умолчанию: sqlite:///onboardpro.db)
- `SECRET_KEY`: Секретный ключ JWT для аутентификации
- `TELEGRAM_BOT_TOKEN`: Токен Telegram Bot API
- `GOOGLE_CREDENTIALS_PATH`: Путь к файлу учетных данных Google API
- `WORKABLE_API_KEY`: Ключ API Workable для интеграции с рекрутингом

Переменные окружения frontend:

- `REACT_APP_API_URL`: URL backend API (по умолчанию: http://localhost:8000)

## Backend API

### Эндпоинты аутентификации

- `POST /auth/login`: Вход пользователя
- `POST /auth/refresh`: Обновление JWT токена
- `GET /users/me`: Получение информации о текущем пользователе

### Управление пользователями

- `GET /users`: Список всех пользователей (только для HR)
- `POST /users`: Создание нового пользователя (только для HR)
- `PUT /users/{id}`: Обновление пользователя (только для HR)
- `DELETE /users/{id}`: Удаление пользователя (только для HR)

### Управление задачами

- `GET /tasks`: Список задач
- `POST /tasks`: Создание задачи
- `PUT /tasks/{id}`: Обновление задачи
- `DELETE /tasks/{id}`: Удаление задачи

### Управление планами

- `GET /plans`: Список планов адаптации (доступно для HR и менеджеров)
- `POST /plans`: Создание нового плана (только для HR)
- `PUT /plans/{id}`: Обновление плана (только для HR)
- `DELETE /plans/{id}`: Удаление плана (только для HR)

### Обратная связь

- `GET /feedback`: Список обратной связи
- `POST /feedback`: Создание обратной связи
- `PUT /feedback/{id}`: Обновление обратной связи
- `DELETE /feedback/{id}`: Удаление обратной связи

## Схема базы данных

### Таблица Users

- `id`: Первичный ключ
- `email`: Email пользователя (уникальный)
- `password_hash`: Хеш пароля
- `role`: Роль пользователя (HR, Manager, Employee)
- `department`: Отдел пользователя
- `first_name`: Имя пользователя
- `last_name`: Фамилия пользователя
- `middle_name`: Отчество пользователя
- `phone`: Телефон пользователя в формате +380 XX XXX XX XX (обновление от 28.04.2025)
- `disabled`: Флаг блокировки пользователя (true/false)
- `photo`: Ссылка на фотографию пользователя (обновление от 28.04.2025)
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Tasks

- `id`: Первичный ключ
- `title`: Название задачи
- `description`: Описание задачи
- `user_id`: ID назначенного пользователя (внешний ключ)
- `plan_id`: ID плана (внешний ключ)
- `priority`: Приоритет задачи (низкий, средний, высокий)
- `status`: Статус задачи
- `deadline`: Срок выполнения
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Plans

- `id`: Первичный ключ
- `title`: Название плана
- `description`: Описание плана
- `role`: Целевая роль (сотрудник, менеджер)
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Feedback

- `id`: Первичный ключ
- `content`: Содержание обратной связи
- `from_user`: ID пользователя-отправителя (внешний ключ)
- `to_user`: ID пользователя-получателя (внешний ключ)
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

## Миграции базы данных

### Использование Alembic

Для управления схемой базы данных используется Alembic:

- **Инициализация миграций**: Миграции уже инициализированы в проекте
- **Конфигурация**: Настройки находятся в файле `alembic.ini`
- **Создание миграций**: `alembic revision --autogenerate -m "описание изменений"`
- **Применение миграций**: `alembic upgrade head`
- **Откат миграций**: `alembic downgrade -1`
- **История миграций**: `alembic history`

### Структура миграций

- `backend/migrations/env.py`: Конфигурация окружения миграций
- `backend/migrations/versions/`: Директория с файлами миграций
- `backend/alembic.ini`: Основной конфигурационный файл Alembic

### Механизмы устойчивости баз данных

Для обеспечения стабильной работы с базой данных реализованы следующие механизмы:

- **Проверка структуры БД при запуске**: Скрипт `start.sh` проверяет наличие необходимых таблиц и полей
- **Прямое обновление БД**: Скрипт `direct_db_update.py` для аварийного обновления структуры БД при проблемах с миграциями
- **Многоуровневый подход обновления**:
  1. Стандартный процесс миграции через Alembic
  2. Прямое обновление БД в случае неудачи миграций
  3. Аварийное создание недостающих таблиц и полей средствами SQLite

### Валидация соответствия моделей и БД

Инструмент `validate_db_models.py` позволяет:

- Проверить, соответствует ли структура БД моделям SQLAlchemy
- Выявить недостающие колонки и таблицы
- Получить рекомендации по исправлению проблем

- **Анализ результатов**: `backend/analyze_load_test.py` для анализа логов и создания отчетов

- Время отклика (медиана, 95-й, 99-й процентили)
- Запросов в секунду (RPS)
- Количество успешных/неуспешных запросов

- Пропускная способность сети

## Устранение неполадок

### Распространенные проблемы

- **Переменные окружения**: Убедитесь, что все необходимые переменные установлены в `.env`
- **Подключение к базе данных**: Проверьте переменную окружения DATABASE_URL
- **Аутентификация**: Проверьте секретный ключ JWT
- **Сборка Frontend**: Проверьте npm зависимости
- **Проблемы с Docker**: Проверьте установку Docker и docker-compose

### Проблемы с авторизацией и работой с фотографиями

- **Ошибка 500 при авторизации**: Возможно, структура базы данных не соответствует модели:
  1. Проверьте логи сервера на наличие ошибок типа `no such column`
  2. Запустите скрипт `validate_db_models.py` для проверки соответствия моделей и БД
  3. При необходимости перезапустите контейнеры с `docker-compose down && docker-compose up -d`
  4. В случае повторяющихся проблем используйте ручное обновление базы: `python direct_db_update.py`
- **Проблемы с загрузкой фотографий**:
  1. Убедитесь, что директория `static/photos` существует и доступна для записи
  2. Проверьте, что поле `photo` корректно определено в модели и существует в таблице
  3. При необходимости примените миграцию для добавления поля: `alembic upgrade head`
  4. При проблемах с доступом к фотографиям проверьте правильность URL изображений (Исправлено 28.04.2025)
  5. Если URL изображений начинается с `http://backend:8000/`, это указывает на проблему с внутренними Docker именами (Исправлено 28.04.2025)

### Проблемы с обновлением данных

- **Операции с базой данных не отражаются в интерфейсе**:
  1. Проверьте корректность API запросов в консоли браузера
  2. Используйте кнопки обновления данных из базы (добавлено 28.04.2025) для синхронизации
  3. Проверьте правильность обработки ответов API на фронтенде
  4. При необходимости перезагрузите страницу для принудительного обновления данных
