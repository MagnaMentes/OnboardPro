## Архитектура системы

OnboardPro следует современной архитектуре веб-приложения:

- **Backend**: FastAPI (Python) с базой данных SQLite и SQLAlchemy ORM
- **Frontend**: React с Tailwind CSS и Heroicons
- **Инфраструктура**: Docker и docker-compose для контейнеризации
- **Аутентификация**: JWT-based система аутентификации
- **Внешние интеграции**: Telegram Bot, Google Calendar, Workable API
- **Миграции**: Alembic для управления версиями базы данных

## Среда разработки

### Предварительные требования

- Docker и Docker Compose
- Node.js (v18+) и npm
- Python 3.11+

### Инструкции по установке

1. Клонировать репозиторий
2. Создать файл `.env` с необходимыми переменными окружения
3. Запустить `docker-compose up -d` для старта контейнеров
4. Открыть приложение по адресу http://localhost:3000

### Переменные окружения

Необходимые переменные окружения:

- `DATABASE_URL`: Путь к базе данных SQLite (по умолчанию: sqlite:///onboardpro.db)
- `SECRET_KEY`: Секретный ключ JWT для аутентификации
- `TELEGRAM_BOT_TOKEN`: Токен Telegram Bot API
- `GOOGLE_CREDENTIALS_PATH`: Путь к файлу учетных данных Google API
- `WORKABLE_API_KEY`: Ключ API Workable для интеграции с рекрутингом

Переменные окружения frontend:

- `REACT_APP_API_URL`: URL backend API (по умолчанию: http://localhost:8000)

## Backend API

### Эндпоинты аутентификации

- `POST /auth/login`: Вход пользователя
- `POST /auth/refresh`: Обновление JWT токена
- `GET /users/me`: Получение информации о текущем пользователе

### Управление пользователями

- `GET /users`: Список всех пользователей (только для HR)
- `POST /users`: Создание нового пользователя (только для HR)
- `PUT /users/{id}`: Обновление пользователя (только для HR)
- `DELETE /users/{id}`: Удаление пользователя (только для HR)

### Управление задачами

- `GET /tasks`: Список задач
- `POST /tasks`: Создание задачи
- `PUT /tasks/{id}`: Обновление всех полей задачи (только для HR, добавлено 29.04.2025)
- `PUT /tasks/{id}/status`: Обновление статуса задачи
- `DELETE /tasks/{id}`: Удаление задачи

### Управление планами

- `GET /plans`: Список планов адаптации (доступно для HR и менеджеров)
- `POST /plans`: Создание нового плана (только для HR)
- `PUT /plans/{id}`: Обновление плана (только для HR)
- `DELETE /plans/{id}`: Удаление плана (только для HR)

### Обратная связь

- `GET /feedback`: Список обратной связи
- `POST /feedback`: Создание обратной связи
- `PUT /feedback/{id}`: Обновление обратной связи
- `DELETE /feedback/{id}`: Удаление обратной связи

#### Механизм управления кэшированием данных аналитики (добавлено 01.05.2025)

- **На стороне сервера**:

  - Система версионирования для отслеживания актуальности данных в кэше
  - Глобальная версия аналитики для обеспечения согласованности всех данных
  - Автоматическая инвалидация кэша при изменении данных задач
  - Поддержка параметра `refresh=true` для принудительного обновления данных
  - Дополнительные метаданные для каждого ответа с временной меткой и информацией о версии
  - Сокращенное время хранения данных в кэше (до 2-5 секунд) для быстрого обновления

- **На стороне клиента**:
  - Модифицированные запросы с уникальным timestamp для обхода кэша браузера
  - HTTP-заголовки для отключения кэширования: `Cache-Control`, `Pragma`, `Expires`
  - Увеличенная задержка (3 сек) перед запросом обновлений для гарантии получения актуальных данных
  - Метки метаданных для отслеживания источника изменений (локальные vs. серверные)
  - "Умное" объединение данных с приоритезацией наиболее актуальных значений
  - Специальные механизмы для первой загрузки страницы с принудительным обновлением

## WebSocket и обновления в реальном времени

### Архитектура WebSocket

Для обеспечения обновлений в реальном времени наше приложение использует WebSocket-соединения. Основная функциональность реализована в файле `websocket_manager.py`.

#### Жизненный цикл WebSocket соединения

1. **Установка соединения** - когда пользователь авторизуется, фронтенд устанавливает WebSocket-соединение с сервером.
2. **Аутентификация** - токен JWT передается в параметрах запроса при подключении WebSocket.
3. **Регистрация соединения** - соединение добавляется в список активных соединений для конкретного пользователя.
4. **Отправка и получение сообщений** - происходит обмен сообщениями между клиентом и сервером.
5. **Закрытие соединения** - когда пользователь выходит из системы или закрывает вкладку, соединение закрывается.

#### Приоритезация сообщений

Система поддерживает различные типы сообщений с разным уровнем приоритета:

| Тип сообщения         | Приоритет         | Описание                                     |
| --------------------- | ----------------- | -------------------------------------------- |
| `task_status_changed` | 1 (высокий)       | Обновление статуса задачи                    |
| `analytics_update`    | 2 (очень высокий) | Обновление аналитических данных              |
| `notification`        | 0 (обычный)       | Уведомления пользователя                     |
| `ping`                | -                 | Проверка активности соединения (вне очереди) |

Очередь сообщений обрабатывает сначала высокоприоритетные сообщения, обеспечивая быструю доставку наиболее важной информации.

#### Надежность доставки сообщений

Для обеспечения надежной доставки сообщений используется:

- Асинхронная очередь сообщений с приоритетами
- Блокировки для синхронизации доступа к очереди
- Механизм принудительной обработки высокоприоритетных сообщений
- Повторные попытки отправки при обнаружении ошибки

```python
async def process_message_queue(self, force: bool = False):
    """Обрабатывает очередь сообщений для отправки клиентам"""
    # Если обработчик уже запущен и нет принудительной обработки, выходим
    if not force and self.queue_processor_running:
        return

    self.queue_processor_running = True

    try:
        while True:
            # Проверяем, есть ли сообщения в очереди
            if not self.message_queue:
                # Если очередь пуста и нет принудительной обработки, завершаем цикл
                if force:
                    break
                # Иначе ждем новых сообщений
                await asyncio.sleep(0.1)
                continue

            # Получаем сообщение из очереди с блокировкой
            message = None
            async with self.queue_lock:
                if self.message_queue:
                    # Сортируем сообщения по приоритету, если их несколько
                    if len(self.message_queue) > 1:
                        # Сортируем по приоритету (больше - выше приоритет)
                        messages = list(self.message_queue)
                        messages.sort(key=lambda x: x.get("priority", 0), reverse=True)
                        message = messages[0]
                        self.message_queue.remove(message)
                    else:
                        message = self.message_queue.popleft()

            if message:
                await self.send_queued_message(message)

            # Небольшая задержка для предотвращения блокировки цикла событий
            await asyncio.sleep(0.01)

            if force:
                break

    finally:
        # Сбрасываем флаг только если не было принудительной обработки
        if not force:
            self.queue_processor_running = False
```

#### Обработка ошибок

Система включает многоуровневую обработку ошибок:

- Отслеживание состояния WebSocket соединения перед отправкой сообщений
- Логирование всех ошибок в файл `logs/websocket_manager.log`
- Автоматическое отключение неактивных соединений
- Восстановление после критических ошибок

### Система аналитики в реальном времени

#### Кэширование аналитических данных

Для оптимизации производительности используется многоуровневая система кэширования:

1. **В памяти сервера** - основной кэш хранит результаты аналитических запросов
2. **Версионирование кэша** - каждое обновление данных обновляет глобальную версию кэша
3. **Адаптивное время жизни кэша** - время кэширования зависит от текущей нагрузки на сервер

```python
def invalidate_analytics_cache():
    """
    Инвалидирует кеш аналитики путем обновления глобальной версии.
    Эта функция должна вызываться при любых изменениях, которые влияют на аналитические данные.
    """
    global _analytics_global_version
    global _analytics_cache

    # Полностью очищаем кеш вместо инкремента версии
    _analytics_cache.clear()

    # Увеличиваем глобальный счетчик версий для принудительного обновления данных
    _analytics_global_version += 1

    # Выводим лог для отладки кеширования
    print(f"[DEBUG] Аналитический кеш инвалидирован. Новая версия: {_analytics_global_version}")

    # Возвращаем новую версию
    return _analytics_global_version
```

#### Синхронизация аналитических данных

Обновление аналитических данных происходит при следующих событиях:

1. Изменение статуса задачи
2. Создание новой задачи
3. Удаление задачи
4. Обновление плана онбординга

При любых изменениях, затрагивающих аналитику, происходит:

1. Полная инвалидация кэша аналитики
2. Отправка WebSocket-сообщения о изменении конкретной сущности
3. Небольшая задержка для завершения обработки основных изменений
4. Генерация и отправка свежих аналитических данных через WebSocket

#### Обновление UI в реальном времени

На стороне фронтенда обновления обрабатываются компонентами:

1. `HRDashboard.jsx` - основной компонент панели HR с аналитикой
2. `WebSocketContext.jsx` - провайдер контекста для WebSocket-соединений
3. `useWebSocket.js` - хук для удобного использования WebSocket в компонентах

При получении обновления аналитики через WebSocket, компоненты проверяют метаданные сообщения, чтобы правильно обновить соответствующие части интерфейса.

### Инструменты диагностики WebSocket соединений

Для отладки WebSocket соединений можно использовать:

1. **Логи сервера**:

   ```
   logs/websocket_manager.log
   ```

2. **Эндпоинт статистики WebSocket**:  
   `GET /api/websocket/stats` - возвращает информацию о текущих соединениях и статистике сообщений

3. **Консоль браузера**: мониторинг WebSocket соединений через Network -> WS

4. **Отладка через консоль**:
   ```javascript
   // В консоли браузера
   window.__debug_websocket = true; // Включает расширенное логирование WebSocket
   ```

## Устранение неполадок

### Распространенные проблемы

- **Переменные окружения**: Убедитесь, что все необходимые переменные установлены в `.env`
- **Подключение к базе данных**: Проверьте переменную окружения DATABASE_URL
- **Аутентификация**: Проверьте секретный ключ JWT
- **Сборка Frontend**: Проверьте npm зависимости
- **Проблемы с Docker**: Проверьте установку Docker и docker-compose

### Проблемы с авторизацией и доступом к страницам

- **Перенаправление на страницу логина при попытке доступа к защищенной странице**:
  1. Проверьте наличие токена в localStorage с помощью инструментов разработчика
  2. Убедитесь, что токен не истек и является валидным
  3. Проверьте соответствие роли пользователя требуемым правам доступа для страницы
  4. При необходимости выполните повторный вход в систему
  5. Если проблема повторяется, очистите localStorage и кэш браузера

### Проблемы с отображением графиков и визуализаций

- **Графики отображаются некорректно или не отображаются**:
  1. Проверьте наличие данных для отображения в консоли браузера
  2. Убедитесь, что компонент `AnalyticsChart` получает корректные пропсы
  3. Проверьте соответствие структуры данных требуемому формату для Chart.js
  4. При проблемах с производительностью уменьшите количество точек данных с помощью параметра `maxPoints`

### Проблемы с обновлением данных

- **Операции с базой данных не отражаются в интерфейсе**:
  1. Проверьте корректность API запросов в консоли браузера
  2. Используйте кнопки обновления данных из базы (добавлено 28.04.2025) для синхронизации
  3. Проверьте правильность обработки ответов API на фронтенде
  4. При необходимости перезагрузите страницу для принудительного обновления данных

### Проблемы с функциональностью drag-and-drop в календаре

- **Ошибка "Не удалось обновить дату задачи" при перетаскивании события** (исправлено 29.04.2025):

  1. Проверьте формат данных, отправляемых на сервер в функции `handleEventDrop` в компоненте CalendarView.jsx
  2. Убедитесь, что используется поле `deadline` вместо устаревшего `due_date`
  3. Проверьте наличие всех обязательных полей модели `TaskCreate` в запросе на обновление:
     - `plan_id`
     - `user_id`
     - `title`
     - `description`
     - `priority`
  4. Убедитесь, что дата отправляется в формате ISO: `event.start.toISOString()`
  5. Проверьте в консоли ответ сервера на запрос обновления задачи
  6. При необходимости перезапустите приложение: `docker-compose restart`

- **События не перетаскиваются или обновление даты происходит некорректно**:
  1. Проверьте наличие атрибута `editable: true` в настройках календаря
  2. Убедитесь, что компонент FullCalendar правильно настроен с активированными опциями:
     ```javascript
     editable={true}
     droppable={true}
     eventDrop={handleEventDrop}
     ```
  3. Проверьте консоль браузера на наличие ошибок JavaScript
  4. Проверьте права доступа текущего пользователя (функциональность может быть ограничена для некоторых ролей)
  5. Перезагрузите страницу для обновления состояния календаря
