## Архитектура системы

OnboardPro следует современной архитектуре веб-приложения:

- **Backend**: FastAPI (Python) с базой данных SQLite и SQLAlchemy ORM
- **Frontend**: React с Tailwind CSS и Heroicons
- **Инфраструктура**: Docker и docker-compose для контейнеризации
- **Аутентификация**: JWT-based система аутентификации
- **Внешние интеграции**: Telegram Bot, Google Calendar, Workable API
- **Миграции**: Alembic для управления версиями базы данных

## Среда разработки

### Предварительные требования

- Docker и Docker Compose
- Node.js (v18+) и npm
- Python 3.11+

### Инструкции по установке

1. Клонировать репозиторий
2. Создать файл `.env` с необходимыми переменными окружения
3. Запустить `docker-compose up -d` для старта контейнеров
4. Открыть приложение по адресу http://localhost:3000

### Переменные окружения

Необходимые переменные окружения:

- `DATABASE_URL`: Путь к базе данных SQLite (по умолчанию: sqlite:///onboardpro.db)
- `SECRET_KEY`: Секретный ключ JWT для аутентификации
- `TELEGRAM_BOT_TOKEN`: Токен Telegram Bot API
- `GOOGLE_CREDENTIALS_PATH`: Путь к файлу учетных данных Google API
- `WORKABLE_API_KEY`: Ключ API Workable для интеграции с рекрутингом

Переменные окружения frontend:

- `REACT_APP_API_URL`: URL backend API (по умолчанию: http://localhost:8000)

## Backend API

### Эндпоинты аутентификации

- `POST /auth/login`: Вход пользователя
- `POST /auth/refresh`: Обновление JWT токена
- `GET /users/me`: Получение информации о текущем пользователе

### Управление пользователями

- `GET /users`: Список всех пользователей (только для HR)
- `POST /users`: Создание нового пользователя (только для HR)
- `PUT /users/{id}`: Обновление пользователя (только для HR)
- `DELETE /users/{id}`: Удаление пользователя (только для HR)

### Управление задачами

- `GET /tasks`: Список задач
- `POST /tasks`: Создание задачи
- `PUT /tasks/{id}`: Обновление задачи
- `DELETE /tasks/{id}`: Удаление задачи

### Управление планами

- `GET /plans`: Список планов адаптации
- `POST /plans`: Создание нового плана
- `PUT /plans/{id}`: Обновление плана
- `DELETE /plans/{id}`: Удаление плана

### Обратная связь

- `GET /feedback`: Список обратной связи
- `POST /feedback`: Создание обратной связи
- `PUT /feedback/{id}`: Обновление обратной связи
- `DELETE /feedback/{id}`: Удаление обратной связи

## Frontend реализация

### Структура компонентов

- `App.js`: Главный компонент приложения
- `components/`: Переиспользуемые UI компоненты
- `pages/`: Компоненты страниц
  - `ManagerDashboard.jsx`: Интерфейс менеджера с двумя основными разделами:
    - Управление задачами: Создание и управление задачами через модальные формы
    - Управление планами: Создание и управление планами адаптации через модальные формы
- `context/`: React контекст провайдеры
- `hooks/`: Пользовательские React хуки
- `utils/`: Утилиты

### UI/UX дизайн

- **Модальные окна**: Формы для создания задач и планов отображаются в модальных окнах
- **Организация разделов**: Контент организован в логические разделы с четкими заголовками
- **Адаптивный дизайн**:
  - Десктоп (≥1280px): Полная навигация с иконками и текстом
  - Планшет (≥768px, <1280px): Навигация только с иконками
  - Мобильный (<768px): Навигация через бургер-меню
- **Визуальная иерархия**:
  - Четкие заголовки разделов
  - Единый стиль кнопок
  - Цветовая индикация статуса
  - Интуитивные таблицы

### Управление состоянием

- React Context API для глобального состояния
- Локальное состояние компонентов для UI элементов:
  - Состояния видимости модальных окон
  - Состояния форм
  - Состояния загрузки и ошибок
- Управление состоянием форм через контролируемые компоненты

## Схема базы данных

### Таблица Users

- `id`: Первичный ключ
- `email`: Email пользователя (уникальный)
- `password_hash`: Хеш пароля
- `role`: Роль пользователя (HR, Manager, Employee)
- `department`: Отдел пользователя
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Tasks

- `id`: Первичный ключ
- `title`: Название задачи
- `description`: Описание задачи
- `user_id`: ID назначенного пользователя (внешний ключ)
- `plan_id`: ID плана (внешний ключ)
- `priority`: Приоритет задачи (низкий, средний, высокий)
- `status`: Статус задачи
- `deadline`: Срок выполнения
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Plans

- `id`: Первичный ключ
- `title`: Название плана
- `description`: Описание плана
- `role`: Целевая роль (сотрудник, менеджер)
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

### Таблица Feedback

- `id`: Первичный ключ
- `content`: Содержание обратной связи
- `from_user`: ID пользователя-отправителя (внешний ключ)
- `to_user`: ID пользователя-получателя (внешний ключ)
- `created_at`: Временная метка создания
- `updated_at`: Временная метка обновления

## Миграции базы данных

### Использование Alembic

Для управления схемой базы данных используется Alembic:

- **Инициализация миграций**: Миграции уже инициализированы в проекте
- **Конфигурация**: Настройки находятся в файле `alembic.ini`
- **Создание миграций**: `alembic revision --autogenerate -m "описание изменений"`
- **Применение миграций**: `alembic upgrade head`
- **Откат миграций**: `alembic downgrade -1`
- **История миграций**: `alembic history`

### Структура миграций

- `backend/migrations/env.py`: Конфигурация окружения миграций
- `backend/migrations/versions/`: Директория с файлами миграций
- `backend/alembic.ini`: Основной конфигурационный файл Alembic

## Тестирование

### Модульное тестирование

- **Фреймворк**: pytest
- **Директория тестов**: `backend/tests/`
- **Конфигурация**: `backend/pytest.ini`
- **Запуск тестов**: `pytest tests/test_api.py -v`
- **Тестовая БД**: `backend/test.db` (автоматически создается при запуске тестов)

### Нагрузочное тестирование

- **Инструменты**: Python `locust` для создания нагрузки
- **Сценарии**: `backend/load_test.py` содержит сценарии нагрузочного тестирования
- **Запуск**: `./run_load_test.sh` запускает тесты в Docker-контейнере
- **Конфигурация**: `docker-compose.loadtest.yml` содержит настройки для контейнеров тестирования
- **Анализ результатов**: `backend/analyze_load_test.py` для анализа логов и создания отчетов
- **Документация**: Подробная инструкция в `load_testing_guide.md`

#### Метрики нагрузочного тестирования

- Время отклика (медиана, 95-й, 99-й процентили)
- Запросов в секунду (RPS)
- Количество успешных/неуспешных запросов
- Использование CPU и памяти
- Пропускная способность сети

## Развертывание

### Docker развертывание

1. Создать файл `.env` с необходимыми переменными окружения
2. Собрать Docker образы: `docker-compose build`
3. Запустить контейнеры: `docker-compose up -d`
4. Открыть приложение по адресу http://localhost:3000

### Конфигурация контейнеров

- Backend контейнер:
  - Эндпоинт проверки здоровья: `/health`
  - Политика автоматического перезапуска: `unless-stopped`
  - Монтирование томов для разработки
- Frontend контейнер:
  - Сервер разработки на порту 3000
  - Политика автоматического перезапуска: `unless-stopped`
  - Монтирование томов для разработки
  - Том node_modules для производительности

### Продакшн развертывание

1. Настроить продакшн сервер
2. Настроить переменные окружения
3. Собрать и развернуть Docker контейнеры
4. Настроить обратный прокси (рекомендуется Nginx)
5. Настроить SSL сертификаты

## Устранение неполадок

### Распространенные проблемы

- **Переменные окружения**: Убедитесь, что все необходимые переменные установлены в `.env`
- **Подключение к базе данных**: Проверьте переменную окружения DATABASE_URL
- **Аутентификация**: Проверьте секретный ключ JWT
- **Сборка Frontend**: Проверьте npm зависимости
- **Проблемы с Docker**: Проверьте установку Docker и docker-compose
- **Проблемы с авторизацией HR-пользователей**: При возникновении проблем с авторизацией HR-пользователей, выполните перезапуск контейнеров с помощью команды:
  ```
  docker-compose down && docker-compose up -d
  ```
- **Дублирование данных**: При обнаружении дублирования данных в приложении выполните полную пересборку контейнеров с очисткой кэша:
  ```
  docker-compose down
  rm backend/onboardpro.db # удаление существующей базы данных
  docker-compose build --no-cache
  docker-compose up -d
  ```

### Решение проблем с производительностью

- **Медленная загрузка данных**: Проверьте сетевое подключение между контейнерами
- **Высокое использование CPU**: Проанализируйте логи на предмет зацикливания или бесконечной рекурсии
- **Утечки памяти**: Проверьте использование памяти контейнерами: `docker stats`
- **Медленный фронтенд**: Используйте инструменты профилирования React для обнаружения излишних ререндеров

### Логи

- Логи backend: `docker-compose logs backend`
- Логи frontend: `docker-compose logs frontend`
- Детальные логи с отслеживанием: `docker-compose logs -f backend` или `docker-compose logs -f frontend`
- Просмотр логов за последние N строк: `docker-compose logs --tail=100 backend`

### Управление базой данных

- **Просмотр схемы**: `docker-compose exec backend python -c "from models import Base; print([t.__tablename__ for t in Base.__subclasses__()])"` 
- **Полный сброс базы данных**:
  ```
  docker-compose down
  rm backend/onboardpro.db
  docker-compose up -d
  ```
- **Применение миграций вручную**: `docker-compose exec backend alembic upgrade head`
