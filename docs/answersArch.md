## Общее описание проекта

### Основная цель и ценность решения

OnboardPro — это современная платформа для автоматизации и оптимизации процесса адаптации новых сотрудников в компании. Основная цель проекта — сделать процесс онбординга структурированным, эффективным и прозрачным для всех участников.

**Целевая аудитория:**

- **HR-специалисты:** управление процессом адаптации, создание планов онбординга, мониторинг прогресса
- **Руководители отделов/менеджеры:** контроль выполнения задач своих подчиненных, предоставление обратной связи
- **Новые сотрудники:** получение четкого плана действий, задач, необходимых материалов и возможность отслеживать свой прогресс

### Ключевые проблемы и потребности пользователей

OnboardPro решает следующие ключевые проблемы:

1. **Несистематизированный процесс адаптации:**

   - Отсутствие стандартизированных чек-листов и планов адаптации
   - Разрозненность информации и материалов для новых сотрудников
   - Сложность отслеживания прогресса адаптации

2. **Коммуникационные разрывы:**

   - Сложности коммуникации между HR, руководителями и новыми сотрудниками
   - Отсутствие прозрачности в статусе выполнения задач
   - Задержки в получении обратной связи

3. **Низкая эффективность процесса:**
   - Большие временные затраты HR и руководителей на административные задачи
   - Отсутствие аналитики эффективности процесса онбординга
   - Невозможность быстро масштабировать процесс при росте компании

### Реализованные и планируемые функции

**Реализованные функции:**

- Система аутентификации с ролевой моделью (HR, менеджер, сотрудник)
- Управление профилями пользователей с возможностью загрузки фотографий
- Создание и редактирование планов онбординга
- Отслеживание выполнения задач с различными статусами
- Приоритизация задач (высокий, средний, низкий приоритет)
- Фильтрация и поиск задач
- HR-дашборд с базовой аналитикой
- Интеграции с Telegram (уведомления), Google Calendar (синхронизация событий) и Workable (импорт данных о сотрудниках)

**Планируемые функции:**

- Расширение аналитического модуля с более глубокими метриками эффективности онбординга
- Внедрение системы оценки качества адаптации на основе обратной связи
- Расширение интеграций с HR-системами и инструментами совместной работы
- Система шаблонов для быстрого создания типовых планов онбординга
- Мобильное приложение для удобства использования вне офиса
- Масштабирование инфраструктуры для поддержки крупных корпоративных клиентов

### Уникальные особенности проекта

1. **Комплексный подход к онбордингу** — объединение административных, обучающих и социальных аспектов адаптации
2. **Гибкость настройки процессов** — возможность создания индивидуальных планов адаптации с учетом специфики должности и отдела
3. **Интеграционные возможности** — связь с популярными HR-системами и корпоративными инструментами
4. **Аналитический компонент** — сбор и анализ данных об эффективности онбординга для постоянного улучшения процесса
5. **Адаптивный пользовательский интерфейс** — удобство использования на различных устройствах

## Текущая архитектура

### Общая архитектура приложения

OnboardPro построен как монолитное приложение с разделением на серверную и клиентскую части, связанные через API. Такая архитектура была выбрана для ускорения начальной разработки и упрощения развертывания.

**Технологический стек:**

**Фронтенд:**

- React.js — основная библиотека для построения пользовательского интерфейса
- TailwindCSS — фреймворк для стилизации компонентов
- PostCSS — инструмент для трансформации CSS с плагинами
- Heroicons — коллекция SVG-иконок

**Бэкенд:**

- FastAPI — высокопроизводительный веб-фреймворк для создания API на Python
- SQLAlchemy — ORM для работы с базой данных
- Alembic — инструмент для управления миграциями базы данных
- Pydantic — библиотека для валидации данных и парсинга

**База данных:**

- SQLite — в текущей версии для упрощения разработки и развертывания
- (Планируется миграция на PostgreSQL для продакшена)

**Инфраструктура:**

- Docker — контейнеризация приложения
- Docker Compose — оркестрация контейнеров
- (CI/CD в процессе настройки, вероятно будет использоваться GitHub Actions)

### Организация фронтенда

Фронтенд построен на React с функциональными компонентами и хуками. Структура организована по следующему принципу:

```
frontend/
├── public/           # Статические файлы
└── src/              # Исходный код
    ├── components/   # Переиспользуемые компоненты
    │   ├── common/   # Общие компоненты (кнопки, формы, модальные окна)
    │   ├── layout/   # Компоненты макета (шапка, меню, футер)
    │   └── specific/ # Специфичные компоненты для конкретных страниц
    ├── pages/        # Компоненты страниц
    ├── config/       # Конфигурационные файлы
    ├── hooks/        # Пользовательские React-хуки
    ├── services/     # Сервисы для работы с API
    ├── utils/        # Утилитарные функции
    └── context/      # React контексты для глобального состояния
```

Для управления состоянием используется комбинация React Context API и локального состояния компонентов. Redux не используется для упрощения разработки, учитывая текущие масштабы приложения.

### Бэкенд и API

Бэкенд реализован на FastAPI, что обеспечивает высокую производительность и удобство разработки благодаря автоматической генерации документации API.

**Основные API-эндпоинты:**

- `/users` — управление пользователями
- `/tasks` — управление задачами онбординга
- `/plans` — управление планами онбординга
- `/integrations` — интеграции с внешними сервисами
- `/stats` — получение статистики и аналитики

API реализовано по принципам REST. Каждый эндпоинт поддерживает соответствующие HTTP-методы (GET, POST, PUT, DELETE) и возвращает ответы в формате JSON. Аутентификация реализована через JWT-токены.

**Документация API:**

- Автоматически генерируемая Swagger UI доступна по адресу `/docs`
- ReDoc доступен по адресу `/redoc`

### База данных

В текущей версии используется SQLite для простоты разработки и развертывания. Основные таблицы в базе данных:

1. `users` — информация о пользователях системы

   - id, email, password_hash, role, first_name, last_name, middle_name, department, phone, photo, disabled

2. `tasks` — задачи онбординга

   - id, title, description, status, priority, due_date, created_at, updated_at, user_id, plan_id

3. `plans` — планы онбординга

   - id, title, description, created_at, updated_at, creator_id, department

4. `feedback` — обратная связь по задачам

   - id, task_id, user_id, content, rating, created_at

5. `notifications` — уведомления для пользователей
   - id, user_id, content, type, read, created_at

Схема данных нормализована и включает необходимые связи между таблицами (один ко многим, многие ко многим).

### Интеграции с внешними сервисами

В приложении реализованы следующие интеграции:

1. **Telegram Bot API**

   - Отправка уведомлений пользователям в Telegram
   - Напоминания о задачах и дедлайнах
   - Настраивается через `TELEGRAM_BOT_TOKEN` в .env

2. **Google Calendar API**

   - Синхронизация задач онбординга с календарем
   - Создание событий для встреч и важных дат
   - Настраивается через сервисный аккаунт (credentials.json)

3. **Workable API**
   - Импорт данных о новых сотрудниках из HR-системы Workable
   - Настраивается через `WORKABLE_API_KEY` в .env

Интеграции реализованы модульно в файле `integrations.py` с возможностью легкого подключения новых сервисов в будущем.

### Инфраструктура

Приложение разворачивается с помощью Docker и Docker Compose, что обеспечивает изоляцию и простоту развертывания.

**Среды развертывания:**

- Локальная разработка — через Docker Compose или напрямую на хост-машине
- (Планируется) Тестовый сервер — AWS EC2 или аналогичный сервис
- (Планируется) Продакшен — вероятно, AWS с использованием Elastic Beanstalk или ECS

CI/CD находится в процессе настройки, предполагается использование GitHub Actions для автоматизации тестирования и деплоя.

## UI/UX компонент

### Подход к дизайну интерфейса

OnboardPro использует современный минималистичный дизайн с акцентом на юзабилити и четкую визуальную иерархию. Основные принципы:

1. **Консистентность** — единый стиль для всех компонентов и страниц
2. **Адаптивность** — корректная работа на устройствах с различными размерами экранов
3. **Доступность** — соблюдение базовых принципов a11y для инклюзивности

**Используемые инструменты и библиотеки:**

- TailwindCSS как основа для стилизации
- Компонентный подход с повторным использованием UI-элементов
- Heroicons для иконографики
- Нативные HTML-элементы с кастомной стилизацией вместо сложных компонентных библиотек для лучшей производительности

### Основные пользовательские сценарии

#### 1. Процесс авторизации и начало работы

- Пользователь заходит на страницу логина
- После успешной авторизации перенаправляется на дашборд в соответствии со своей ролью
- Для нового пользователя может активироваться процесс заполнения профиля

#### 2. Сценарий для HR-специалиста

- Создание нового плана онбординга (определение задач, сроков, ответственных)
- Назначение плана новому сотруднику
- Мониторинг прогресса через HR-дашборд
- Получение и анализ обратной связи
- Управление пользователями (создание, блокировка, сброс паролей)

#### 3. Сценарий для менеджера

- Просмотр задач для своих подчиненных
- Отметка о выполнении задач, добавление комментариев
- Создание дополнительных задач при необходимости
- Просмотр аналитики по своему отделу

#### 4. Сценарий для сотрудника

- Просмотр персонального плана онбординга
- Отметка о выполнении задач
- Запрос помощи или уточнений по задачам
- Предоставление обратной связи о процессе адаптации

### Реализованные элементы интерфейса

Основные страницы и компоненты:

1. **Аутентификация**

   - Страница логина с обработкой ошибок
   - Форма сброса пароля

2. **Дашборды**

   - HR-дашборд с метриками и аналитикой
   - Дашборд менеджера с задачами по сотрудникам
   - Персональный дашборд сотрудника

3. **Управление профилями**

   - Список пользователей с возможностью фильтрации
   - Создание/редактирование профилей
   - Загрузка фотографий пользователей
   - Управление статусами (активация/деактивация)

4. **Управление задачами**

   - Список задач с фильтрацией по статусу и приоритету
   - Создание/редактирование задач
   - Отслеживание прогресса выполнения

5. **Управление планами онбординга**

   - Создание шаблонов планов
   - Назначение планов сотрудникам
   - Отслеживание прогресса выполнения планов

6. **Интеграции**
   - Настройка интеграций с внешними сервисами
   - Управление уведомлениями

Большинство страниц реализовано с модальными окнами для создания/редактирования сущностей, что обеспечивает единообразие интерфейса.

### Сильные и слабые стороны UX

**Сильные стороны:**

- Интуитивно понятная навигация
- Консистентный дизайн модальных окон и форм
- Отзывчивый интерфейс с быстрой обратной связью
- Адаптивность для различных устройств
- Четкая визуальная иерархия

**Стороны, требующие доработки:**

- Ограниченная кастомизация для клиентов с различными потребностями
- Отсутствие расширенных фильтров и поиска на некоторых страницах
- Недостаточная оптимизация для мобильных устройств на некоторых страницах
- Ограниченные возможности для массовых операций
- Отсутствие режима работы офлайн

## Кодовая база и разработка

### Организация кода и репозитория

Код хранится в репозитории GitHub. Основные ветки:

- `main` — стабильная версия для production
- `develop` — основная ветка для разработки
- Создаются feature-ветки для отдельных задач в формате `feature/название-задачи`

**Процесс код-ревью:**

- Pull-реквесты создаются из feature-веток в develop
- Обязательный код-ревью от минимум одного разработчика
- Автоматические проверки линтеров и тестов
- Мердж только после успешного прохождения всех проверок и код-ревью

**Процесс деплоя:**

- Автоматический деплой на тестовую среду из ветки develop
- Ручной запуск деплоя на продакшен из ветки main
- Использование тегов для маркировки релизов

### Инструменты и практики разработки

**Линтеры и форматтеры:**

- ESLint и Prettier для JavaScript/React
- Black и flake8 для Python
- pre-commit хуки для проверки при коммите

**Тестирование:**

- Модульные тесты для бэкенда
- Нагрузочное тестирование API

**Документация:**

- JSDoc для документирования JavaScript
- Docstrings в формате Google для Python
- Автоматическая генерация документации API через Swagger

**Контейнеризация:**

- Docker для изоляции окружения
- Docker Compose для оркестрации

### Покрытие тестами

**Бэкенд:**

- Модульные тесты с помощью pytest
- Тесты API-эндпоинтов
- Нагрузочное тестирование с использованием Python и Locust

**Фронтенд:**

- Базовые тесты React-компонентов (планируется расширение)

Текущее покрытие тестами оценивается примерно в 60-70% для бэкенда и существенно меньше для фронтенда.

### Проблемные части кода

1. **Фронтенд:**

   - Некоторый дублирующийся код в компонентах страниц
   - Недостаточная типизация (отсутствие TypeScript)
   - Неоптимальное управление состоянием для сложных форм
   - Несистематичная обработка ошибок

2. **Бэкенд:**

   - Некоторые эндпоинты имеют слишком много ответственности
   - Недостаточная валидация входных данных в некоторых местах
   - Не все эндпоинты полностью покрыты тестами
   - Использование SQLite может стать ограничением при масштабировании

3. **Интеграции:**
   - Отсутствие механизма повторных попыток при сбоях интеграций
   - Недостаточное логирование и мониторинг интеграционных процессов

### Использование GitHub Copilot

GitHub Copilot активно используется в проекте для следующих задач:

- Генерация шаблонного кода (например, CRUD-операций)
- Помощь в написании документации
- Рефакторинг повторяющегося кода
- Предложение оптимизаций и улучшений
- Помощь в написании тестов

Особенно полезен Copilot при работе с FastAPI и React, где он хорошо понимает структуру кода и предлагает релевантные решения.

Roadmap проекта ведется в Notion. Основные приоритеты на ближайшие 3-6 месяцев:

1. **Приоритет 1 (Критический):**

   - Миграция с SQLite на PostgreSQL для улучшения производительности и надежности
   - Улучшение механизмов безопасности (двухфакторная аутентификация, улучшение политик паролей)
   - Оптимизация производительности фронтенда

2. **Приоритет 2 (Высокий):**

   - Разработка API для мобильного приложения
   - Расширенный аналитический модуль
   - Интеграция с дополнительными HR-системами
   - Внедрение механизма уведомлений в реальном времени (WebSockets)

3. **Приоритет 3 (Средний):**
   - Реализация шаблонов для планов онбординга
   - Персонализация интерфейса для различных ролей
   - Экспорт данных в различные форматы (PDF, Excel)
   - Локализация интерфейса (поддержка нескольких языков)

### Ключевые вызовы и блокеры

1. **Технические вызовы:**

   - Масштабирование базы данных при увеличении числа пользователей

### 2. Технические детали

#### 2.1. Безопасность

- **HTTPS:**
  - FastAPI настроен для работы с HTTPS через Nginx в production
  - В development-окружении используется HTTP
  - Все API-запросы выполняются через защищенное соединение в production

- **Хеширование паролей:**
  - Используется passlib с bcrypt для хеширования паролей
  - Реализована проверка сложности паролей (минимум 8 символов, спецсимволы, цифры)

- **Меры безопасности:**
  - Защита от XSS через правильную обработку пользовательского ввода
  - CSRF-токены для защиты форм
  - Параметризованные запросы через SQLAlchemy для предотвращения SQL-инъекций
  - Валидация входных данных с использованием Pydantic

- **Планы по улучшению:**
  - Реализация 2FA через SMS или authenticator apps
  - Внедрение механизма блокировки аккаунта после нескольких неудачных попыток входа
  - Усиление политик паролей с историей использования
  - Реализация механизма регулярной смены паролей

#### 2.2. SQLite

- **Текущий объем данных:**
  - ~500 пользователей в тестовой базе
  - ~2000 задач
  - ~200 планов онбординга
  - Общий размер базы: ~10 МБ

- **Узкие места в запросах:**
  - Проблема N+1 при загрузке задач с вложенными связями:
    ```python
    # Пример проблемного запроса
    tasks = db.query(Task).all()
    for task in tasks:
        # Это вызывает дополнительный запрос для каждой задачи
        user = task.user
        plan = task.plan
    ```

  - Решение: использование `joinedload` для оптимизации:
    ```python
    tasks = db.query(Task).options(
        joinedload(Task.user),
        joinedload(Task.plan)
    ).all()
    ```

- **Индексы:**
  - Созданы базовые индексы для первичных и внешних ключей
  - **Таблицы, требующие оптимизации:**
    - `tasks` (требуются индексы по полям status, priority, due_date)
    - `users` (требуются индексы по полям role, department)
    - `notifications` (требуются индексы по полям user_id, read)

#### 2.3. Асинхронные задачи

- **Уведомления:**
  - Базовая асинхронная обработка реализована с использованием asyncio для Telegram-уведомлений
  - **Не реализовано:**
    - Email-уведомления
    - Push-уведомления в браузере

- **Очереди:**
  - В текущей версии очереди не используются
  - Планируется использование python-rq для обработки асинхронных задач, особенно для:
    - Создания и рассылки отчетов
    - Массовых уведомлений
    - Синхронизации с внешними системами

#### 2.4. Проблемные зоны

- **Перегруженные эндпоинты:**
  - `/tasks` имеет слишком много ответственности:
    - Создание/обновление/удаление задач
    - Фильтрация и сортировка
    - Управление статусами
    - Обработка связанных сущностей (комментарии, вложения)
  - Рекомендация: разделить на несколько специализированных эндпоинтов или использовать GraphQL

- **Дублирующийся код фронтенда:**
  - Формы создания/редактирования имеют много повторяющейся логики валидации
  - Компоненты таблиц дублируют код для сортировки/фильтрации
  - Пример: `TaskForm.js` и `PlanForm.js` имеют схожие компоненты для выбора пользователей и дат

- **Технический долг:**
  - Отсутствие TypeScript для типизации на фронтенде
  - Недостаточное покрытие тестами
  - Неконсистентная обработка ошибок в API
  - Отсутствие стратегии кэширования данных
  - Устаревшие зависимости, требующие обновления
  - Недостаточно гибкая система ролей (требуется более детальное разграничение прав)

  - Для создания документации: может помочь технический писатель (удаленно, частичная занятость)

### 5. Прочее

#### 5.1. Баги, ограничения и пожелания

- **Неучтенные ограничения:**
  - SQLite имеет ограничения при одновременном доступе множества пользователей
  - Текущая архитектура не оптимизирована для горизонтального масштабирования
  - Медленная загрузка при большом количестве задач (>1000)

- **Пожелания пользователей из тестовой группы:**
  - Возможность массового редактирования задач
  - Более гибкие настройки уведомлений
  - Темная тема интерфейса
  - Возможность добавления комментариев к задачам

#### 5.2. Сложности со стеком

- **FastAPI:** в целом стабилен, но есть сложности с:
  - Оптимальной структурой больших приложений (разделение на модули)
  - Правильной обработкой исключений в асинхронном коде
  - Интеграцией с некоторыми библиотеками, ожидающими синхронный код

- **React:** основные сложности связаны с:
  - Управлением состоянием в глубоко вложенных компонентах
  - Оптимизацией перерендеринга при частом обновлении данных
  - Отсутствием TypeScript для типизации пропсов и состояний

- **SQLite:** 
  - Ограничения при конкурентных запросах
  - Отсутствие некоторых продвинутых SQL-функций
  - Потенциальные проблемы с производительностью при росте базы данных
